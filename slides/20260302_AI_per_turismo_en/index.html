AI in Tourism: Overview

Definitions · State of the Art · Common Myths · The Paradigm Shift

Sec. 1 · Overview

AI, ML, Deep Learning, Generative AI

  • AI — systems that perform tasks typically done by humans
  • ML — learns from data, without hand-written rules
  • Deep Learning — ML with deep neural networks
  • Generative AI — produces text, images, audio, code
Sec. 1 · Overview

AI in Tourism: where we stand today

Hotels & OTAs

  • Predictive revenue management
  • Booking chatbots
  • Automated review responses

Destinations

  • Personalised itineraries
  • Tourist sentiment analysis
  • Flow forecasting

Travel Agencies

  • Automated search and comparison
  • Generated travel proposals
  • After-sales support

Adoption is still uneven: large OTAs are years ahead of SMEs

Sec. 1 · Overview

What AI is not

Myth Reality
"AI is always accurate" It produces convincing errors (hallucinations)
"It understands everything like a human" It predicts the next token — it doesn't reason
"It is autonomous and unpredictable" It follows statistical distributions over data
"It will replace everyone" It automates repetitive tasks, not entire roles
"It is too complex for SMEs" APIs and no-code tools lower the barrier to entry
Sec. 1 · Overview

The Paradigm Shift

Research AI (pre-2020)

  • Specialised models for a single task
  • Required large labelled datasets
  • Accessible only to large companies
  • Output: classifications, numerical predictions

Generative AI (today)

  • Foundation models: one model for hundreds of tasks
  • Instructed via natural language
  • APIs accessible to anyone
  • Output: text, images, code, audio
The barrier to entry has dropped dramatically: a well-written prompt replaces months of ML development.

Section 2

Large Language Models

How large language models work

Sec. 2 · LLM

What is an LLM

A Large Language Model is a neural network trained to predict the next token in a sequence of text.

Input: "The best hotel in Rome for families is" ↓ [LLM: billions of parameters] ↓ Output: "the" → "Grand" → "Hotel" → "Palatino" → ...
  • "Large" = billions of parameters (GPT-4: ~1.8T estimated)
  • "Language" = operates on text (and now also images, audio)
  • "Model" = a mathematical function learned from data
There is no understanding, no intentionality: it is statistical prediction at enormous scale.
Sec. 2 · LLM

Tokenisation: text becomes numbers

Before processing text, the LLM converts it into tokens (word fragments).

"Reservation" → ["Res", "erv", "ation"] → [4921, 1203, 3847] "ChatGPT" → ["Chat", "G", "PT"] → [10314, 38, 2898] "€" → ["€"] → [27975]
  • ~1 token ≈ 0.75 words in English / ~0.6 words in Italian
  • API cost is measured in tokens (input + output)
  • Rare or technical words → more tokens → more expensive

Interactive tool: platform.openai.com/tokenizer

Sec. 2 · LLM

The Transformer architecture (simplified)

Input tokens ↓ [Embedding: token → numeric vector] ↓ [Attention: each token "looks at" all others] ↓ [Feed-forward: non-linear transformation] ↓ × N blocks ↓ [Output: probability distribution over vocabulary] ↓ Token with highest prob → generated text
The self-attention mechanism allows the model to understand context: "bank" in "river bank" vs "bank account".
Sec. 2 · LLM

Pre-training: learning to predict

What happens

  • Input: trillions of words from the internet, books, code
  • Objective: predict the next token
  • Duration: weeks on thousands of GPUs
  • Cost: tens to hundreds of millions of $

What it learns

  • Grammar and syntax
  • Facts about the world
  • Basic logical reasoning
  • Different writing styles
The model "knows" only what was in the training data. It has no knowledge of recent news or proprietary company data.
Sec. 2 · LLM

Post-training: from predictor to assistant

Base model (prediction only) ↓ [Supervised Fine-Tuning on instructions] ↓ Instruction-tuned model (follows commands) ↓ [RLHF: human feedback + reward model] ↓ Aligned model (helpful, safe, honest)
  • SFT — (instruction → ideal response) pairs written by humans
  • RLHF — human raters compare responses, the model learns what they prefer
  • Result: the model "converses" instead of completing sentences
Sec. 2 · LLM

Generation parameters

Parameter Effect Example use
temperature 0 = deterministic, 1 = creative, >1 = chaotic 0.2 for FAQs, 0.9 for copywriting
top_p Truncates to top P% of cumulative probability 0.9 safe default
max_tokens Limit of generated tokens 150 for tweets, 2000 for articles
stop String that stops generation "\n\n" for short responses
For deterministic tasks (classification, data extraction): temperature = 0.

Sec. 2 · LLM

Closed vs open source models

Proprietary (third-party APIs)

  • No infrastructure to manage
  • Automatic updates
  • Variable cost per token
  • Data sent to the provider
  • Example: OpenAI, Anthropic, Google

Open-source / open-weight

  • Full data control
  • Fixed cost (hardware)
  • Requires technical expertise
  • Full customisation
  • Example: LLaMA, Mistral, Phi
For a tourism SME: start with APIs, consider open-source only if you have privacy requirements or very high volumes.
Sec. 2 · LLM

Consumer-grade interfaces vs API

Consumer-grade interfaces (ChatGPT, Claude.ai, Gemini)

  • Manual use, session by session
  • No integration with business systems
  • Suitable for experimentation and individual use
  • Data not used for training (paid plan)

API (OpenAI API, Claude API, etc.)

  • Integrable into apps, CRM, PMS, website
  • Automatable and scalable
  • Requires technical development (or no-code)
  • Pay-per-use billing
Sec. 2 · LLM

How to choose a model: criteria for SMEs

  • Task quality — use top models only if the task is critical (e.g. responses to VIP guests)
  • Cost per call — check the pricing: input and output tokens have different rates
  • Latency — real-time chatbots require fast models (e.g. GPT-4o mini, Claude Haiku)
  • Language support — test explicitly: not all models perform equally in non-English languages
  • Data compliance — verify DPA, data residency (EU), terms of use
Practical rule: use the smallest/cheapest model that solves your problem.

Context Window

The model's working memory: limits and practical implications

Sec. 3 · Context Window

What is the context window

The context window is the maximum amount of text the model can "see" in a single processing moment.

┌─────────────────────────── Context Window ──────────────────────────────┐ │ System prompt │ Previous conversation │ Documents │ Latest question │ └─────────────────────────────────────────────────────────────────────────┘ ↑ everything must fit in here ↑
ModelContext window≈ English words
GPT-4o128,000 tokens~96,000
Claude 3.5 Sonnet200,000 tokens~150,000
Gemini 2 Pro2,000,000 tokens~1,500,000
GPT-4o mini128,000 tokens~96,000
Sec. 3 · Context Window

What happens when the context is full

  • The model cannot process more text: the API call returns an error
  • Some implementations "trim" the oldest parts of the conversation
  • The model "forgets" the trimmed information: no native persistent memory
  • Response quality degrades if the context is very long (lost-in-the-middle)
A 100-page PDF can easily saturate 128k token contexts. Always check before sending long documents.
Sec. 3 · Context Window

Managing context in real applications

Software strategies

  • Progressive summarization — summarize the chat every N turns
  • Sliding window — keep only the last K exchanges
  • RAG — retrieve only the relevant pieces (→ section 5)
  • Entity extraction — save important facts to a DB

Cost implications

  • Input tokens are charged on every call
  • Long conversation = growing cost
  • Keeping the system prompt short matters
  • Models with caching (Claude) reduce costs

Sec. 3 · Context Window

The trend: growing context windows

  • 2023: GPT-4 with 8k → 32k tokens
  • 2024: Claude with 200k, Gemini with 1M tokens
  • 2025: Gemini 2.0 with 2M tokens (~1,500 pages)
  • Direction: "unlimited" context window in the long term
Larger contexts do not eliminate the need for RAG: attention quality on long texts remains an open problem, and cost is proportional.
Do not confuse "can process 1M tokens" with "processes them accurately": always validate on your specific use case with appropriate benchmarks.

Fine-tuning

Training a model on your own data

Sec. 4 · Fine-tuning

The adaptation continuum

Zero-shot Few-shot Fine-tuning │ │ │ ▼ ▼ ▼ "Classify "Here are 3 examples, Train the model this now classify" with thousands of review" my own examples │ │ │ No data A few examples Dataset required Immediate in the prompt Time + cost
For most tourism use cases, prompting and few-shot are sufficient. Fine-tuning only when you have a concrete problem that cannot be solved otherwise.
Sec. 4 · Fine-tuning

Training types compared

Phase Data Who does it Cost
Pre-training Trillions of tokens from the internet Labs only (OpenAI, Anthropic…) $10M–$100M+
Supervised FT A few hundred to thousands of examples Companies, developers $100–$10,000
RLHF / RLAIF Human or AI preferences Mainly the labs High
Prompting Examples in the text Anyone API call cost
Sec. 4 · Fine-tuning

Full fine-tuning vs LoRA

Full fine-tuning

  • Updates all model parameters
  • Maximum adaptation capacity
  • Requires high-end GPUs
  • Risk of "forgetting" base knowledge (catastrophic forgetting)

LoRA / PEFT

  • Adds small adapters to the base model
  • Trains only a fraction of parameters (<1%)
  • Consumer-grade GPUs are sufficient
  • The base model remains intact
  • The dominant technique today for local fine-tuning
Sec. 4 · Fine-tuning

When fine-tuning is worth it

  • The model must adopt a very specific communication style (brand tone, register)
  • Repetitive task with a rigid output format (e.g. structured JSON, specific forms)
  • Long prompt that repeats every time: fine-tuning "bakes in" the instructions
  • You have hundreds of high-quality examples already validated
Do not fine-tune to teach the model new information: use RAG for that. Fine-tuning teaches how to respond, not what to know.
Sec. 4 · Fine-tuning

Requirements: data, costs, infrastructure

Data

  • Min. 50–100 examples (quality > quantity)
  • Format: JSONL with prompt/completion
  • Must be clean and representative

Cost (API)

  • OpenAI FT: ~$8 / 1M training tokens
  • One-time training + ongoing inference
  • Model hosting may have a fixed cost

Infrastructure

  • Via API: no infrastructure needed
  • Self-hosted: GPU with 16–80 GB VRAM
  • Cloud ML: Vertex AI, SageMaker, RunPod
Sec. 4 · Fine-tuning

Example: specialised chatbot for a hotel

Without fine-tuning

Prompt: "You are the Hotel Bellavista assistant.
Reply in a warm and professional manner.
Here are our policies: [500 tokens]
Here are our services: [800 tokens]
Guest question: late check-out?"

~1,400 tokens of context on every call

With fine-tuning

Prompt: "Guest question:
late check-out?"

Tone, policies and style already baked into the model

~30 tokens of context on every call

Savings on input tokens = direct long-term savings on API costs.
Sec. 4 · Fine-tuning

Fine-tuning vs prompting: the choice

Criterion Use prompting Consider fine-tuning
Time available Available immediately Weeks for data + training
Request volume Low / medium Very high (context cost)
Output style Generic or semi-specific Very rigid or branded
Available data Few or none Tens to hundreds of examples
Technical skills None required Developer or ML engineer

Section 5

RAG — Retrieval Augmented Generation

Connecting an LLM to your data without training it

Sec. 5 · RAG

The problem: the LLM doesn't know your data

What the LLM doesn't know

  • Your updated price list
  • Your hotel's specific policies
  • Real-time availability
  • Internal company documents
  • News from the past few months

The (bad) alternatives

  • Put everything in the prompt → context explodes, high costs
  • Fine-tuning → doesn't learn updatable facts, only style
  • Ignore the problem → the model hallucinates
RAG = retrieve the relevant documents at question time, insert them into the context, generate the answer.
Sec. 5 · RAG

RAG Architecture — Core concept

Sec. 5 · RAG

RAG Architecture — Application diagram

[Business documents] ──→ [Chunking] ──→ [Embedding] ──→ [Vector DB] │ │ (indexing, once) │ User question ──→ [Question embedding] ──→ [Similarity search] │ Top-K relevant chunks │ ▼ [Prompt = question + relevant chunks] ──→ [LLM] ──→ Answer
Sec. 5 · RAG

Embeddings: text as coordinates

An embedding model converts text into a vector of numbers that represents its meaning.

"room with sea view" → [0.12, -0.87, 0.45, 0.33, ...] "ocean view bedroom" → [0.11, -0.85, 0.46, 0.31, ...] ← similar! "breakfast included" → [0.78, 0.23, -0.91, 0.55, ...] ← different
  • Texts with similar meaning → nearby vectors in the space
  • Cosine similarity measures how semantically related two texts are
  • Embedding models: OpenAI text-embedding-3, Cohere, sentence-transformers
Sec. 5 · RAG

Embeddings: the vector space

Sec. 5 · RAG

Embeddings: word vectors

Sec. 5 · RAG

Vector database: what it is and how it works

Traditional database

Searches for exact words

SELECT * FROM faq
WHERE text LIKE '%check-out%'

Does not find "departure time"

Vector database

Searches by semantic meaning

query = embed("what time do I need to leave the room?")
results = db.search(query, top_k=3)

Finds check-out, departure, late check-out…

Common vector database solutions: Pinecone, Weaviate, Chroma, pgvector (PostgreSQL), Qdrant

Sec. 5 · RAG

RAG pipeline: step by step

  1. Document loading — PDF, Word, database, website
  2. Chunking — split into pieces of 200–500 tokens with overlap
  3. Embedding — convert each chunk into a vector
  4. Indexing — save the vectors to the vector DB
  5. — runtime —
  6. Question embedding — convert the user question into a vector
  7. Search — find the top-3/5 most similar chunks
  8. Augmentation — insert the chunks into the prompt
  9. Generation — the LLM responds based on the chunks
Sec. 5 · RAG

RAG in tourism: practical examples

Dynamic hotel FAQ

  • Operations manual → chunked
  • Chatbot answers on policies, hours, services
  • Update: upload new PDF

Tour catalogue

  • 500 tour sheets → indexed
  • "Tour suitable for children with snorkeling" → top-3 tours
  • Response with precise details

Prices and availability

  • Nightly update from the PMS
  • Chatbot responds with fresh data
  • No risk of outdated information
Sec. 5 · RAG

RAG vs fine-tuning: comparison table

Aspect RAG Fine-tuning
Data updates Easy (reload documents) Expensive (retrain)
Transparency High (source is visible) Low (data baked in)
Factual hallucinations Reduced (chunks as anchor) Present
Output style Does not change model behaviour Modifies style and behaviour
Technical setup Medium (vector DB, pipeline) Medium-high (dataset, GPU)
In practice the two are often combined: fine-tuning for style + RAG for up-to-date data.
Sec. 5 · RAG

RAG limitations

  • Chunking quality — poorly split chunks = fragmented context = wrong answer
  • Multi-hop questions — "compare the price of tour X with tour Y" requires multiple retrievals
  • Non-textual data — complex tables, images, PDFs with complex layout are difficult
  • Added latency — vector search adds 50–300ms per query
  • Hallucination not eliminated — the model can still ignore the provided context
RAG is not a magic wand: it requires careful engineering and continuous evaluation of response quality.

Section 6

AI Agents

Autonomous systems that plan and act in the real world

Sec. 6 · AI Agents

The difference: LLM, Assistant, Agent

🧠
Pure LLM
  • Predicts the next token
  • No memory
  • No tools
Input → Output
💬
Assistant
  • Responds in natural language
  • Maintains context
  • Follows system instructions
Input → Output
🤖
Agent
  • Plans complex tasks
  • Uses tools and executes actions
  • Observes results and iterates
Goal → Plan → Actions → Goal
An agent uses an LLM as its reasoning engine, but has operational autonomy.
Sec. 6 · AI Agents

The ReAct cycle: Reason → Act → Observe

Goal: "Find the cheapest flight Rome-Tokyo for July and add it to the calendar" [Reason] → I need to search for flights in July [Act] → search_flights("Rome", "Tokyo", "July") [Observe] → {results: [{price:650, date:"Jul 5"}, ...]} [Reason] → The cheapest is €650 on July 5th [Act] → create_calendar_event("July 5", "Tokyo flight") [Observe] → {status: "created", id: "evt_123"} [Reason] → Task completed [Output] → "Found flight €650 (Jul 5) and created the event"
Sec. 6 · AI Agents

Planning and memory in agents

Memory types

  • In-context — everything in the current conversation
  • External — databases, files, persistent vectors
  • Procedural — instructions in the system prompt
  • Episodic — logs of previous sessions

Planning types

  • Chain-of-thought — step by step in the text
  • Tree-of-thought — explores multiple paths
  • Plan-and-execute — full plan then execution
  • ReAct — interleaves reasoning and actions
Sec. 6 · AI Agents

Multi-agent systems

👤 "Plan a weekend in Florence for 2 people, budget €400"
↓
🎯 Orchestrator
↓
✈️ Flights Agent
API Skyscanner
↓
🏨 Hotel Agent
PMS + Booking API
↓

Key concepts

  • Specialisation — each agent has a precise domain and dedicated tools
  • Orchestration — the orchestrator assigns tasks, collects results and resolves conflicts
  • Parallelism — multiple agents work simultaneously, reducing latency
  • Composability — sub-agents are reusable in other workflows

Frameworks

  • LangGraph — agent graphs with explicit state
  • CrewAI — agents with roles and goals in natural language
  • AutoGen — multi-agent conversations (Microsoft)
  • Claude MCP — standard protocol for tools and agents
Sec. 6 · AI Agents

AI Agents in tourism

🏨 Automated booking

Receives request, checks PMS availability, applies rate policies, generates confirmation and sends email to the guest

💰 Revenue optimisation

Analyses history, local events and competitors → automatically proposes the weekly rate plan to the revenue manager

🔍 Competitor monitoring

Scans OTAs every hour, compares rates with your own, notifies the team if the gap exceeds a defined threshold

⭐ Review management

Monitors TripAdvisor, Google, Booking in real time, drafts personalised replies and sends them after human approval

🗺️ Traveller assistant

Plans tailor-made itineraries, books restaurants and activities, answers pre- and post-stay questions in natural language

📊 Operations reports

Aggregates data from PMS, CRM and OTA channels every morning → generates KPI report with highlighted anomalies, ready for the briefing

Agents in production require human oversight on high-impact decisions (e.g. rate changes, public replies, sending communications).
Sec. 6 · AI Agents

Risks and limitations of agents

  • Cascading errors — a mistake in one step is amplified in subsequent steps
  • Infinite loops — the agent does not terminate if it cannot find the stop condition
  • Prompt injection — malicious external data crafted to hijack the agent's behaviour
  • Unpredictable costs — every iteration consumes tokens: complex tasks cost a lot
  • Irreversible actions — sending emails, modifying databases, spending money
Rule: every irreversible action must have an explicit human approval checkpoint.
Sec. 6 · AI Agents

When are agents reliable today?

Scenario Reliability Notes
Simple tasks with clear tools High 1-3 steps, well-defined tools
Information search and synthesis High No irreversible actions
Multi-step workflow (5-10 steps) Medium Requires supervision
Open-ended tasks, ambiguous goal Low Not recommended in production
Autonomous financial actions Not recommended Human supervision always

Section 7

Tools and Function Calling

How an LLM executes actions in the real world

Sec. 7 · Tools

From the LLM to the real world

An LLM on its own only produces text. To do something, it needs tools.

User: "What is the availability of the suite on August 15th?" LLM recognises: I need to call the "check_availability" tool ↓ Generates structured JSON: {"tool": "check_availability", "params": {"room": "suite", "date": "2026-08-15"}} ↓ The application code executes the call to the PMS ↓ The result comes back to the LLM as text ↓ LLM formulates the final response to the user
Sec. 7 · Tools

Function calling: defining tools for the LLM

Each tool is defined with a schema that the model uses to generate correct calls.

{
  "name": "check_availability",
  "description": "Checks room availability in the PMS",
  "parameters": {
    "room_type": {
      "type": "string",
      "enum": ["standard", "deluxe", "suite"],
      "description": "Requested room type"
    },
    "arrival_date": {
      "type": "string",
      "format": "date",
      "description": "Arrival date (YYYY-MM-DD)"
    },
    "nights": {
      "type": "integer",
      "description": "Number of nights"
    }
  },
  "required": ["room_type", "arrival_date", "nights"]
}

Sec. 7 · Tools

MCP: Model Context Protocol

MCP is an open standard (Anthropic, 2024) for connecting LLMs to tools and data sources in an interoperable way.

Before MCP

  • Every integrator builds their own format
  • Tools not reusable across different models
  • Multiplied maintenance

With MCP

  • Standard MCP server for each system
  • Works with Claude, GPT-4, Gemini…
  • Ecosystem of ready-made servers (Google Drive, GitHub, databases…)
MCP is still being adopted but represents the direction the industry is moving towards.
Sec. 7 · Tools

Classic function calling: context overhead

Example: web search + article generation (3 sources) [LLM] → {"tool":"web_search", "query":"AI news"} ← round-trip 1 [LLM] ← {urls: [url_1, url_2, url_3]} → ctx +tok_0 [LLM] → {"tool":"web_fetch", "url": url_1} ← round-trip 2 [LLM] ← {content: "…full article…"} → ctx +tok_1 [LLM] → {"tool":"web_fetch", "url": url_2} ← round-trip 3 [LLM] ← {content: "…full article…"} → ctx +tok_2 [LLM] → {"tool":"web_fetch", "url": url_3} ← round-trip 4 [LLM] ← {content: "…full article…"} → ctx +tok_3 [LLM] → {"tool":"write_blog", "sources": [c_1, c_2, c_3]} ← round-trip 5
  • Every intermediate result enters the context: the model must process each response before proceeding
  • With N sources: 2N + 1 round-trips, context usage O(N)
  • Saturated context → performance degradation (lost-in-the-middle); latency and cost grow linearly
Sec. 7 · Tools

Programmatic Tool Calling

Classic function calling

→ web_search("AI news") [ctx +tok_0] ← {urls: [...]} → web_fetch(url_1) [ctx +tok_1] ← {content: "..."} → web_fetch(url_2) [ctx +tok_2] ← {content: "..."} → write_blog(c_1, c_2, c_3) 5 round-trips · all content in context

Programmatic Tool Calling

# The model generates this code:
results = web_search("AI news")

content_list = []
for r in results:
    content = web_fetch(r.url)
    content_list.append(content)

write_blog(content_list)

1 round-trip · only the final output re-enters the LLM context

Instead of emitting a JSON for each tool call, the model generates code (Python / TypeScript) in which tools are callable functions. The code is executed in the execution environment; intermediate results reside in local variables, not in the model's context. Available from Claude Opus 4.5 and Sonnet 4.5 onwards.
Sec. 7 · Tools

Code Execution: enabling and impact

How to enable it (Anthropic API)

tools = [
  # 1. Add the execution environment
  {"type": "code_execution_20250522",
   "name": "code_execution"},

  # 2. Mark tools callable from code
  {
    "name": "query_database",
    "description": "...",
    "input_schema": {...},
    "allowed": ["code_execution"]
  }
]

The code_execution tool provides the sandbox; allowed exposes the function as callable in the environment.

Unlocked capabilities

  • Batch processing — for loops over N elements, executed in parallel
  • Conditional tool selection — if/else on the result of a tool to decide the next tool
  • Local data filtering — transformation and aggregation of results before inserting them into context
  • Round-trip reduction — from 2N+1 calls to 1 (request) + 1 (final response)
Measured reduction: 37–50% of tokens for equivalent tasks.
Anthropic benchmark: context at equivalent task approximately 4× lower compared to classic function calling.
Sec. 7 · Tools

Advanced optimisations for high-efficiency agents

Dynamic Web Filtering

  • The system automatically filters the content of web pages retrieved during web search
  • Only semantically relevant sections for the query enter the context
  • Input token reduction: 24% on average (range 33–62% depending on content)
  • Activation: add the web_search tool — filtering is applied automatically

Tool Search

  • With many MCP servers, loading all definitions into context is prohibitive
  • A special search_tool searches for relevant definitions on demand, via regex or BM25 query
  • Savings: ~95% of context occupied by tool definitions
  • Deferred loading: the definition is loaded only when needed and removed afterwards, freeing space

Tool Use Examples

  • Add input examples to the tool definition (few-shot approach at schema level)
  • Five examples are sufficient in most cases
  • Accuracy in correct tool invocation: 70% → 90%
  • Particularly useful for tools with many parameters or complex structured inputs
The three optimisations are orthogonal and cumulative: combining them simultaneously reduces latency, token consumption and error rate in agents with a high number of tools.

Sec. 9 · Prompting

Anatomy of a prompt: 5 components

┌──────────────────────────────────────────────────────────┐ │ ROLE "You are a tourism marketing expert..." │ ├──────────────────────────────────────────────────────────┤ │ CONTEXT "The hotel is called Bellavista, 4-star, │ │ family-oriented, Amalfi Coast..." │ ├──────────────────────────────────────────────────────────┤ │ TASK "Write a reply to the review below" │ ├──────────────────────────────────────────────────────────┤ │ FORMAT "Max 120 words, warm tone, no jargon" │ ├──────────────────────────────────────────────────────────┤ │ CONSTRAINTS "Do not promise refunds. Do not mention │ │ competitors." │ └──────────────────────────────────────────────────────────┘
You don't always need all 5: add only what is needed. A longer prompt is not always better.
Sec. 9 · Prompting

Zero-shot, few-shot, chain-of-thought

Zero-shot

No examples, instruction only

"Classify this review
as positive or negative:
'The breakfast was cold'"

Fast, cheap, sufficient for simple tasks

Few-shot

2–5 examples before the task

"Positive: 'Breathtaking view'
Negative: 'Small room'
Classify: 'Cold breakfast'"

Improves precision and output format

Chain-of-thought

Ask for explicit reasoning

"Reason step by step:
1) Identify the problem
2) Assess the tone
3) Give the classification"

Complex tasks, reasoning, calculations

Common mistakes in prompting

Error Effect Solution
Prompt too vague Generic, unusable output Specify format, length, tone
No business context Generic, non-branded response Add name, sector, values
Only saying what NOT to do The model doesn't know what to do Say WHAT you want, not just what to avoid
Multiple tasks in one prompt Mediocre result on everything One prompt = one task
No output structure Unpredictable format Specify: "reply as a list", "use JSON"
Sec. 9 · Prompting

Systematic prompt debugging

1. Write the initial prompt ↓ 2. Test with 3–5 different inputs ↓ 3. Identify where it fails ├─ Output too long? → add "max X words" ├─ Wrong tone? → specify the role more clearly ├─ Wrong format? → add an example (few-shot) └─ Missing information? → add context ↓ 4. Change ONE element at a time ↓ 5. Retest, compare ↓ 6. Save the working prompt in the template library

Sec. 9 · Prompting

Prompt Engineering vs. Context Engineering

Source: anthropic.com/engineering/…context-engineering

Prompt Engineering

Optimising the text of a single prompt: role, instruction, format, constraints, few-shot examples.

Context Engineering

Designing the entire context passed to the model:

  • Tools — which tools are available and with what definitions
  • Retrieved documents — RAG chunks, web results, project files
  • Conversation history — which turns to keep or compress
  • Structured outputs — previous tool call results in the context
The conceptual leap: output quality depends more on what is in the context than on the wording of the prompt. Especially relevant for agentic systems.

Security and Privacy

GDPR, practical risks and responsible AI use

Sec. 10 · Security

GDPR and AI: obligations and responsibilities

Relevant GDPR principles

  • Data minimisation — use only necessary data
  • Purpose limitation — do not reuse data for other purposes
  • Accountability — you are the data controller
  • Privacy by design — design with privacy in mind

What changes with AI

  • Data sent to the API may be used for training (check the terms)
  • Automated decisions require transparency (Art. 22)
  • Guest profiling → legal basis required
  • Data breach includes leaks via prompts
Sec. 10 · Security

Specific risks: what can go wrong

Risk Description Impact
Prompt injection A malicious input alters the model's behaviour Exposed data, unauthorised actions
Data leakage The model "remembers" sensitive data inserted in the context Exposure of client/business data
Model inversion Attacks that extract data from model parameters Relevant only for fine-tuning on sensitive data
Hallucinations in factual data The model invents prices, policies, legal information Reputational damage, disputes
Sec. 10 · Security

Practical rules: what to never include in a prompt

Never include:
  • Personally identifiable data (name + email + phone together)
  • Credit card numbers, banking details, identity documents
  • Access credentials (passwords, API keys, tokens)
  • Health data or sensitive guest preferences
  • Trade secrets or confidential contracts
If in doubt: anonymise first. "Guest X booked…" → "A guest booked…"

For batch processing of client data: always use pseudonymisation and keep the mapping locally.

Sec. 10 · Security

AI services with enterprise guarantees

Enterprise options

  • OpenAI Enterprise / API — data not used for training, SOC 2
  • Azure OpenAI — EU data residency, tenant isolation, Microsoft DPA
  • Anthropic Claude API — data not used for training, robust privacy policy
  • Google Vertex AI — data residency, GDPR compliance

What to verify

  • Is data used for training? (no for paid plans)
  • Where is data processed? (data residency)
  • Is a DPA (Data Processing Agreement) available?
  • Certifications: SOC 2, ISO 27001, GDPR?
Free, consumer-grade versions (ChatGPT free) do not offer the same guarantees as paid API plans.
Sec. 10 · Security

Checklist: safe AI use in a tourism SME

  • I have signed a DPA with the AI provider
  • Shadow AI policy: staff must not include personal client data in prompts
  • I use paid plans only (no training on data)
  • I have updated the data processing register
  • Privacy policy updated (mention of AI tools)
  • AI outputs reviewed by a human before sending
  • No API keys in website / app code
  • Access to AI tools restricted to necessary users
  • Auditing: logs of critical AI operations retained
  • Data breach response plan updated
